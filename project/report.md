## 6. Архитектура решения и сервис

Опишите, как из модели получился работающий сервис:

1. **Архитектура пайплайна:**
   - Все скрипты, связанные с данными хранятся в `trashnet/data`.  
   Там лежат функции, которые загружают существующий датасет в DataLoader, которые преобразуют изображение в Tensor для inference.
   - Обработанные данные поступают в модель.
   Функции, связанные с загрузкой весов, создания структуры модели, обучения и inference модели лежат в `trashnet/models`.
   - Скрипты из папки `trashnet/service` связывают все это воедино и предоставляют удобный интерфейс для взаимодействия.

2. **API и endpoints:**
   Реализовано 3 endpoints: `/health`, `/predict`, `/train`:
   - `/health` проверяет работоспособность сервиса.
   - `/predict` принимает изображение и возвращает класс и время выполнения.
   - `/train` принимает zip архив с папками изображений в формате ImageFolder и обучает ResNet18 модель.

3. **Технологический стек:**
   - Для ML части использовался фреймворк `Torch` и связанные с ним библиотеки (`torchvision`, `torchmetrics`)
   - Для cli реализации используется `Typer`
   - Для запуска сервиса используется `FastAPI`
   - Для тестов используется `pytest`
   > Полный список зависимостей находится в `pyproject.toml`
   - Можно использовать Docker, для поднятия контейнера нужно выполнить:
   ```bash
   cd project
   docker build -t aie-project .
   docker run -p 8000:8000 aie-project
   ```

## 8. Ограничения и дальнейшая работа

Будьте честны:
- У проекта есть много ограничений связанных с:
   * Отсутствием проверки всех сценариев (не все ошибки могут быть "отловлены")
   * Отсутствует настройка через конфигурации. Путь к локальной модели зашита в скрипт
   * Точность модели сильно ограничена из-за небольшого и довольно специфичного датасета (мусор редко бывает на чистом белом фоне)
   * Хлипкая структура скриптов, проблемы с путями, модулями
- Если бы было больше времени, я бы сделал:
   * Более сильную модель, собрав большее количество "полевых" данных и протестировав большее количество конфигураций
   * Возможность настройки через внешние конфигурации
   * Больше эндпоинтов (как минимум `/test`)
   * Более корректная и чистая обработка сценариев/ошибок
   * Более чистый и надежный код
