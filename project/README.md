## 2. Структура проекта

Проект организован в следующей структуре:

- `requirements.txt` – зависимости проекта (библиотеки Python, необходимые для запуска).
- `report.md` – отчёт по проекту (постановка задачи, данные, эксперименты, результаты).
- `self-checklist.md` – чеклист самопроверки проекта перед сдачей.
- `notebooks/` – экспериментальные ноутбуки:
  - EDA, прототипы моделей, отдельные эксперименты.
- `src/` – основной код проекта:
  - модули для работы с данными;
  - обучение моделей;
  - запуск сервиса/скриптов.
- `data/` – демонстрационные/учебные данные:
  - **только открытые, синтетические или обезличенные данные;**
  - при необходимости – небольшие выборки для быстрого запуска.
- `configs/` – конфигурационные файлы:
  - `.yaml`, `.json`, `.env.example` и т.п.
- `tests/` – тесты (юнит-тесты, простые проверки).
- `artifacts/` – сохранённые модели, отчёты, результаты экспериментов.

Если структура проекта отличается от базовой, опишите это здесь.

## 3. Требования и установка

### 3.1. Требования

- Python `>= 3.11` (уточните при необходимости).
- Установленные системные зависимости: 
  * `git` для клонирования репозитория
  * `uv` для удобства запуска проекта (опционально)
  * `docker` для поднятия проекта на Docker

### 3.2. Установка окружения

#### Если используете просто python:

```bash
# Перейти в папку проекта
cd project

# Создать виртуальное окружение (опционально, но рекомендуется)
python -m venv .venv

# Активировать окружение:
# Windows:
.venv\Scripts\activate
# Linux / macOS:
source .venv/bin/activate

# Установить зависимости
pip install --upgrade pip
pip install . # файл pyproject.toml должен обязательно находиться в проекте
```

#### Если используете uv:

```bash
# Перейти в папку проекта
cd project

# Создать виртуальное окружение и скачать все зависимости
uv sync
```

## 4. Как запустить проект

Опишите, как именно воспроизвести основные сценарии использования проекта.

Примеры (заполните под свой случай):

### 4.1. Работа с cli 

#### 4.1.1 Проверить работоспособность

python
```bash

python -m trashnet.service.cli check

```
uv
```bash
cd project
uv sync      # при необходимости
uv run trashnet-cli predict "путь/к/файлу.png"
```

#### 4.1.3 Запуск обучения

```bash

```
uv
```bash
cd project
uv sync      # при необходимости
uv run trashnet-cli resnet18 "путь/к/данным" "путь/для/сохранения/модели"
```

> **ВСЕ ДОСТУПНЫЕ КОММАНДЫ И АРГУМЕНТЫ К НИМ МОЖНО ПОЛУЧИТЬ ЧЕРЕЗ `--help`**

### 4.2. Запуск сервиса (API/веб-интерфейс)


#### python
```bash
cd project
source .venv/bin/activate      # при необходимости
uvicorn trashnet.service.api:app --reload --port 8000
```

#### uv
```bash
cd project
uv sync      # при необходимости
uv run uvicorn trashnet.service.api:app --reload --port 8000
```

Или, если используется Docker:

```bash
cd project
docker build -t aie-project .
docker run -p 8000:8000 aie-project
```


> Сервис поднимается на порту 8000
> Всего было сделано 3 эндпоинта:
> - `/health` - проверить работоспособность сервиса
> - `/precit` - предсказать класс мусора по изображению 
> - `/train` - обучить модель resnet18 по изображениям из zip архива
>
> Прочитать подробную информацию и протестировать сервис можно:
> - В браузере по адресу <http://127.0.0.1:8000/docs> если запускаете напрямую через uvicorn
> - В браузере по адресу <http://localhost:8000/docs> если запускаете через Docker

## 8. Ограничения и дальнейшая работа

>У проекта есть много ограничений связанных с:
>   * Отсутствием проверки всех сценариев (не все ошибки могут быть "отловлены")

>
>Если бы было больше времени, я бы сделал:

>   * Возможность настройки через внешние конфигурации
>   * Больше эндпоинтов (как минимум `/test`)
>   * Более корректная и чистая обработка сценариев/ошибок

Итоговая оценка за проект выставляется по пятибалльной шкале (2-5).

Ориентиры для оценки:

- **2** – проект не принят:

  - грубые нарушения требований курса;
  - явный плагиат или формальная имитация работы.
- **3** – проект принят, но реализован на базовом уровне:
  - минимальный функционал есть;
  - по чеклисту `self-checklist.md` выполнено **меньше 5** пунктов;
  - эксперименты и документация слабо проработаны.
- **4** – хороший, рабочий проект:
  - сервис запускается по `README.md`, `/predict` использует реальную модель;
  - есть данные, EDA и эксперименты с метриками;
  - структура кода и конфигураций в целом адекватна;
  - по чеклисту выполнено **не менее 5** пунктов.
- **5** – сильный, хорошо проработанный проект:
  - аккуратно реализован сервис и пайплайн;
  - проведены осмысленные эксперименты и обоснован выбор финальной модели;
  - есть базовая наблюдаемость и работа с конфигами/секретами;
  - документация позволяет быстро понять и воспроизвести решение;
  - по чеклисту выполнено **не менее 9** пунктов.

Чеклист `self-checklist.md` служит для самопроверки студента и как подсказка при проверке.

- качество реализации внутри каждого пункта чеклиста;
